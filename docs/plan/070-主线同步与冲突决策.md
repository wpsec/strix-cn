# 070 主线同步与冲突决策

## 目标

把 `strix-cn` 从 `upstream/main` 吸收上游功能的流程固定下来，优先合并上游能力与修复，但不覆盖本分支已经建立的中文化、兼容性和本地适配。遇到无法安全自动决定的冲突时，要求在合并过程中显式引出问题并提交维护者决策，而不是静默处理。

## 范围

- `origin/main` 与 `upstream/main` 的日常同步流程
- 合并前预检、合并中冲突处理、合并后最小验证
- 冲突分级、升级条件和维护者决策单格式
- 当前仓库截至 `2026-07-22` 的已知高风险冲突热点

## 输入

- `origin/main`：本分支维护线
- `upstream/main`：上游主线
- 当前分支的优先保留项：中文文档、中文提示、受限网络兼容、Burp/Caido 适配、本地运行习惯
- 当前分支的优先吸收项：上游安全修复、运行时修复、测试补齐、依赖修正、容器修复

## 变更点

- 默认在临时同步分支执行 `git fetch upstream` + `git merge upstream/main`
- 禁止直接在日常开发分支上硬合上游，避免把冲突扩散到正在进行的开发
- 默认使用 `merge`，不使用会改写本地历史的 `rebase` 作为常规同步手段
- 对同一文件同时改动时，按“行为不回退、能力不丢失、文案不回退”处理
- 不允许用整文件无脑选 `ours` 或 `theirs` 代替分析，尤其是核心逻辑文件

## 执行步骤

1. 建立临时同步分支，避免直接污染维护线。

   ```bash
   git fetch origin upstream --tags
   git switch -c sync/upstream-2026-07-22 origin/main
   ```

2. 先看上游变更范围，再决定需要重点盯防的目录和测试集。

   ```bash
   git log --oneline --left-right origin/main...upstream/main
   git diff --stat origin/main..upstream/main
   ```

3. 在临时分支执行主线合并，保留标准冲突现场。

   ```bash
   git merge upstream/main
   ```

4. 合并后先列出冲突文件，按风险分层处理，不要一上来就处理核心逻辑。

   ```bash
   git status --short
   git diff --name-only --diff-filter=U
   ```

5. 先解决低风险文档和文案冲突，再处理核心逻辑冲突。
   - 文档冲突优先保留本分支中文化结果，再手工吸收上游新增能力说明
   - 中文 CLI/TUI 文案优先保留本分支默认中文，再补回上游新增状态、参数或提示
   - 核心流程、报告链路、代理工具行为必须做三方人工合并，不能偷懒选边

6. 遇到升级条件立即暂停，不继续拍脑袋往前合。
   - 语义冲突无法同时满足双方行为
   - 接口变化会影响现有调用方或测试前提
   - 依赖策略变化会影响打包、运行时、受限网络或本地适配
   - 功能二选一，无法在当前轮次同时保留

7. 冲突处理完成后执行最小验证，确保不是“合上了但功能退了”。
   - 按受影响目录选择最小测试集，例如 `tests/test_ui_localization.py`、`tests/test_proxy_client.py`、`tests/test_report_writer.py`、`tests/test_reporting_fields.py`
   - 需要时补充 `.venv/bin/strix -h` 等冒烟验证，确认 CLI 主路径可用
   - 如果本轮涉及文档以外的行为变更，至少验证对应能力还能启动或生成结果

8. 验证通过后，再把临时同步分支合回维护线。

   ```bash
   git switch main
   git merge --no-ff sync/upstream-2026-07-22
   ```

## 冲突决策规则

| 分级 | 典型范围 | 默认动作 |
|------|------|------|
| 可自动倾向保留本分支 | 中文 `README`、中文 `docs`、中文 CLI/TUI 文案 | 以本分支中文化结果为基线，手工吸收上游新增说明、参数和入口，不允许直接丢弃上游新增内容 |
| 必须三方人工合并 | 核心流程、报告链路、代理交互、代理提示词、代理工具行为 | 对照共同基线逐段合并，确认双方行为都还成立，不能用整文件选边收场 |
| 必须升级给维护者决策 | 语义冲突、接口变化、依赖策略变化、功能二选一、无法同时保留双方行为 | 立即暂停，输出冲突决策单，由维护者拍板后再继续 |

### 保留本分支能力的原则

- 中文文档、中文提示和中文默认输出不回退到英文
- 受限网络兼容、本地运行路径、Burp/Caido 适配不被上游默认实现覆盖
- 上游安全修复、运行时修复、测试补齐和依赖修正优先吸收
- 对同一文件同时改动时，目标是把双方能力叠加起来，而不是只保住某一边

### 提交给维护者决策的固定格式

每个阻塞冲突单独整理一份决策单，格式如下：

```md
## 冲突决策单

- 冲突文件：
- 上游改动摘要：
- 本分支改动摘要：
- 两边不能兼容的点：
- 可选方案 A：
- 可选方案 B：
- 推荐方案：
- 风险：
```
